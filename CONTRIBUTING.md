## How Content Flows Through the System

Understanding this pipeline helps when you need to add or modify content types:

1. **Markdown files** in `src/pages/` contain frontmatter (structured data) and body content
2. **Decap CMS** (`static/admin/config.yml`) defines the editing UI for these files
3. **`gatsby-node.js`** processes markdown into GraphQL nodes, with schema defined in `gatsby/schema/base.gql`
4. **Resolvers** in `gatsby-node.js` provide safe defaults for missing fields (this is where null-handling lives — not in components)
5. **GraphQL fragments** (`src/graphql/fragments.ts`) select fields for page queries
6. **Templates** (`src/templates/`) render pages using data from those queries
7. **Types** are auto-generated by Gatsby on `develop`, then refined in `src/types/index.ts`

## Adding a New Field

1. Add the field to the schema in `gatsby/schema/base.gql`
2. If the field can be missing from content, add a resolver with a default in `gatsby-node.js`
3. Add the field to the relevant fragment in `src/graphql/fragments.ts`
4. Run `npm run develop` to regenerate types
5. Update derived types in `src/types/index.ts` if needed
6. Use the field in the component

## Development Conventions

- **TypeScript strict mode** — all code must compile with `"strict": true`
- **Resolvers over null checks** — handle missing data in the GraphQL resolver layer, not in components
- **Derived types over assertions** — use `NonNullable<>` wrappers from `src/types/index.ts` instead of `!` assertions
- **Accept `readonly`** — Gatsby marks generated types as readonly; don't cast it away
- **Format before committing** — run `npm run format` to keep code style consistent

## Branch and PR Workflow

The main branch is `main`. Netlify deploys automatically on merge.

1. Create a feature branch from `main`
2. Make your changes
3. Run `npm run formatCheck` and `npm run lint` to verify
4. Open a PR against `main`
